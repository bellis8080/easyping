# Story 1.4: Role-Based Access Control (RBAC)

## Status

Approved

## Story

**As a** system administrator,
**I want** users assigned to roles (End User, Agent, Manager, Owner),
**so that** permissions are enforced throughout the application.

## Acceptance Criteria

1. `user_roles` enum created (end_user, agent, manager, owner)
2. Users table includes `role` column (default: end_user)
3. Middleware checks user role before allowing access to agent/manager routes
4. First user created in an organization automatically assigned `owner` role
5. Owner can assign roles to other users via settings page (basic UI)
6. RLS policies updated to respect user roles (e.g., agents can view all tickets, end users only their own)
7. Role checked in UI to show/hide features (e.g., agent dashboard visible only to agents)

### Data Models

**[Source: docs/architecture/data-models.md#user]**

**User Role Enum:**
```typescript
enum UserRole {
  END_USER = 'end_user',   // Can create tickets, view own tickets
  AGENT = 'agent',         // Can view all tickets, respond to tickets
  MANAGER = 'manager',     // Can manage categories, SLA policies, view analytics
  OWNER = 'owner'          // Full admin access, can assign roles
}
```

**Role Hierarchy:**
- **END_USER:** Base role for all users
  - Create tickets
  - View own tickets only
  - Access knowledge base
  - No admin features

- **AGENT:** Support staff role
  - All END_USER permissions
  - View all tickets in organization
  - Respond to tickets
  - Access agent dashboard and analytics
  - Cannot modify categories or policies

- **MANAGER:** Team lead role
  - All AGENT permissions
  - Manage ticket categories
  - Manage SLA policies
  - View detailed analytics and reports
  - Cannot assign roles or manage users

- **OWNER:** Administrator role
  - All MANAGER permissions
  - Assign roles to users
  - Manage AI provider configuration
  - Access user management settings
  - Full system access

**User Interface (Complete):**
```typescript
interface User {

  email: string;
  full_name: string;
  avatar_url: string | null;
  role: UserRole;
  created_at: Date;
  last_seen_at: Date | null;
}
```

### API Specification

**[Source: docs/architecture/api-specification.md]**

**New API Route:**

**PATCH /api/users/:userId/role**

Update a user's role (requires OWNER role).

**Request:**
```typescript
{
  role: 'end_user' | 'agent' | 'manager' | 'owner'
}
```

**Response (Success):**
```typescript
{
  user: {
    id: string;
    email: string;
    full_name: string;
    role: UserRole;
    updated_at: string;
  }
}
```

**Response (Error):**
```typescript
{
  error: {
    code: 'UNAUTHORIZED' | 'FORBIDDEN' | 'NOT_FOUND' | 'VALIDATION_ERROR',
    message: string,
    details?: any
  }
}
```

**Error Codes:**
- `401 UNAUTHORIZED`: Not authenticated
- `403 FORBIDDEN`: Insufficient permissions to assign role
- `404 NOT_FOUND`: User not found
- `400 VALIDATION_ERROR`: Invalid role value

### Critical Implementation Notes

**IMPORTANT:**

1. **Role Hierarchy Enforcement:**
   - Always use `hasPermission(userRole, requiredRole)` helper
   - Never hardcode role strings in conditionals
   - Permission checks must cascade (OWNER can do AGENT tasks)

2. **First User Owner Protection:**
   - First user (OWNER) cannot have role changed via UI
   - Disable role dropdown for first user with tooltip: "Cannot change owner role"
   - If owner role must be transferred, require SQL or Supabase dashboard

3. **Middleware Performance:**
   - Middleware runs on EVERY request
