# Story 1.5: Docker Compose Deployment

## Status

Approved

## Story

**As a** self-hoster,
**I want** to deploy EasyPing with a single `docker-compose up` command,
**so that** I can run it on my own infrastructure easily.

## Acceptance Criteria

1. `docker-compose.yml` includes Supabase services (Postgres, Auth, Storage, Realtime, PostgREST)
2. Next.js app containerized with optimized production Dockerfile
3. Caddy reverse proxy configured for automatic HTTPS (with self-signed cert for local dev)
4. Environment variables externalized to `.env` file
5. Volumes configured for persistent data (database, storage)
6. Health check endpoints for all services
7. `docker-compose up` starts all services successfully on fresh Ubuntu 22.04 VM
8. README includes deployment instructions with system requirements

### Deployment Architecture

**[Source: docs/architecture/deployment-architecture.md, docs/architecture/high-level-architecture.md]**

**Service Architecture:**

```
Internet
  ↓
Caddy Reverse Proxy (Port 80/443)
  ↓
Next.js App Server (Port 3000)
  ↓
Supabase Services:
  - PostgREST API (Port 3001)
  - GoTrue Auth (Port 9999)
  - Storage API (Port 5000)
  - Realtime (Port 4000)
  ↓
PostgreSQL Database (Port 5432)
```

**Docker Networking:**
- All services on default bridge network
- Only Caddy exposes ports to host (80, 443)
- Internal services communicate via service names (e.g., `postgres:5432`)

**Persistent Data:**
- PostgreSQL data: `/var/lib/postgresql/data`
- Supabase Storage: `/var/lib/storage`
- Caddy TLS certificates: `/data/caddy`

**Service Dependencies:**
1. PostgreSQL starts first
2. Supabase services (Auth, Storage, PostgREST, Realtime) depend on PostgreSQL
3. Next.js app depends on Supabase services
4. Caddy depends on Next.js app

**Health Checks:**
- PostgreSQL: `pg_isready -U postgres`
- Next.js: HTTP GET `/api/health`
- All Supabase services: HTTP GET to respective health endpoints

### Docker Multi-Stage Build Strategy

**[Source: docs/architecture/tech-stack.md, docs/architecture/coding-standards.md]**

**Build Optimization:**

**Stage 1: Dependencies**
- Base image: `node:18-alpine`
- Install pnpm globally
- Copy `package.json`, `pnpm-lock.yaml`, `pnpm-workspace.yaml`
- Run `pnpm install --frozen-lockfile`
- Use `turbo prune` to extract minimal dependency tree for web app

**Stage 2: Builder**
- Base image: `node:18-alpine`
- Copy source code from Stage 1
- Copy application source: `apps/web`, `packages/*`
- Run `pnpm build` (builds all workspace dependencies)
- Generate production build in `apps/web/.next`

**Stage 3: Runner (Production)**
- Base image: `node:18-alpine`
- Copy only production dependencies from Stage 1
- Copy built application from Stage 2
- Copy `public` assets and `next.config.js`
- Create non-root user: `nextjs`
- Set working directory: `/app`
- Expose port: `3000`
- Set environment: `NODE_ENV=production`
- Start command: `node apps/web/server.js` (or `next start`)

**Expected Image Size:** ~200-300MB (optimized with Alpine base)

**Layer Caching Strategy:**
- Dependencies layer cached unless `package.json` changes
- Build layer cached unless source code changes
- Runner layer minimal, only includes runtime files

### Caddy Reverse Proxy Configuration

**[Source: docs/architecture/tech-stack.md]**

**Caddyfile Structure:**

```
{$DOMAIN:localhost} {
  # Automatic HTTPS
  tls {$TLS_EMAIL}

  # Reverse proxy to Next.js app
  reverse_proxy web:3000

  # Security headers
  header {
    Strict-Transport-Security "max-age=31536000; includeSubDomains"
    X-Content-Type-Options "nosniff"
    X-Frame-Options "DENY"
    Referrer-Policy "strict-origin-when-cross-origin"
    Permissions-Policy "geolocation=(), microphone=(), camera=()"
  }

  # Enable compression
  encode gzip

  # Logging
  log {
    output stdout
    format json
  }
}
```

**Development Mode (localhost):**
- Caddy auto-generates self-signed certificate for `localhost`
- No email required for local dev
- HTTPS works but browser shows warning (expected)

**Production Mode (real domain):**
- Set `DOMAIN` environment variable to real domain (e.g., `easyping.example.com`)
- Set `TLS_EMAIL` for Let's Encrypt notifications
- Caddy automatically obtains valid Let's Encrypt certificate
- Certificate auto-renews

### Health Check Implementation

**[Source: docs/architecture/high-level-architecture.md]**

**Health Check Endpoint Requirements:**

**GET /api/health**

**Success Response (200 OK):**
```json
{
  "status": "healthy",
  "timestamp": "2025-01-27T10:00:00Z",
  "services": {
    "database": "healthy",
    "auth": "healthy",
    "storage": "healthy",
    "realtime": "healthy"
  },
  "version": "1.0.0"
}
```

**Failure Response (503 Service Unavailable):**
```json
{
  "status": "unhealthy",
  "timestamp": "2025-01-27T10:00:00Z",
  "services": {
    "database": "healthy",
    "auth": "unhealthy",
    "storage": "healthy",
    "realtime": "healthy"
  },
  "error": "Authentication service unavailable"
}
```

**Health Check Logic:**
1. Query database: `SELECT 1` or `SELECT count(*) FROM organizations`
2. Check Supabase Auth: `supabase.auth.getSession()` (should not error)
3. Check Supabase Storage: `supabase.storage.listBuckets()` (should return array)
4. Check Supabase Realtime: Not directly checkable from server-side, rely on service health

**Timeout:** 5 seconds maximum for entire health check

**Docker Healthcheck:**
- Runs every 30 seconds
- Timeout: 3 seconds
- Retries: 3 times before marking unhealthy
- Start period: 10 seconds (grace period on startup)

### Security Considerations

**[Source: docs/architecture/deployment-architecture.md]**

**Secrets Management:**
- All secrets in `.env` file (never commit to Git)
- `.env.example` provides template with placeholder values
- Generate strong passwords: `openssl rand -base64 32`
- JWT secret must be same across GoTrue and PostgREST

**Docker Security:**
- Run Next.js app as non-root user (UID 1001)
- Read-only filesystem where possible
- Drop unnecessary capabilities
- Limit container memory and CPU usage (add resource limits to docker-compose.yml)

**Network Security:**
- Only Caddy exposes ports to host
- Internal services not directly accessible from outside
- PostgreSQL port 5432 not exposed to host (admin can exec into container)
- Optional: Expose PostgreSQL on 54322 for database admin tools (add to AC if needed)

**HTTPS/TLS:**
- Caddy handles HTTPS automatically
- Let's Encrypt certificates in production
- Self-signed certificate for local dev
- Force HTTPS redirects

**Database Security:**
- Strong PostgreSQL password required
- RLS policies enforce tenant isolation
- Connection pooling configured
- Regular backups (automated script provided)

### Critical Implementation Notes

**IMPORTANT:**

1. **Port Configuration:**
   - Local dev (via pnpm): Next.js runs on port **4000**
   - Docker deployment: Next.js runs on port **3000** (standard)
   - Update `next.config.js` to handle both scenarios

2. **Supabase Service Versions:**
   - Use specific versions in `docker-compose.yml` (not `latest`)
   - Avoids breaking changes from automatic updates
   - Documented versions tested and known to work together
   - Check Supabase self-hosting docs for compatible versions: https://supabase.com/docs/guides/self-hosting

3. **Database Migrations:**
   - Migrations stored in `packages/database/migrations`
   - Run migrations manually on first deployment:
     ```bash
     docker-compose run --rm web pnpm db:migrate
     ```
   - Migrations are idempotent (safe to run multiple times)
   - Future: Add migration automation to Docker entrypoint

4. **Environment Variable Precedence:**
   - `.env` file loaded by Docker Compose
   - Environment variables passed to containers
   - Next.js reads `NEXT_PUBLIC_*` variables at build time
   - Need to rebuild Next.js image if public env vars change

5. **Volume Permissions:**
   - PostgreSQL volume owned by `postgres` user (UID 999)
   - Storage volume owned by `storage` user
   - Backup scripts may need `sudo` for volume access
   - Alternative: Use `docker cp` to extract backups

6. **Caddy Data Persistence:**
   - Caddy stores Let's Encrypt certificates in `/data/caddy`
   - MUST persist this volume to avoid hitting Let's Encrypt rate limits
   - Losing volume forces Caddy to re-request certificate (limited to 5 per week)

7. **Development vs Production:**
   - Use `docker-compose.dev.yml` for local development with hot reload
   - Use `docker-compose.yml` alone for production deployment
   - Development mode mounts source code as volumes
   - Production mode copies built artifacts into image
