# Story 2.2: Threaded Conversation in Pings

## Epic Context

This story is part of **Epic 2: Chat-First Ticket Creation & Threading**, which delivers EasyPing's core differentiator - a conversational, chat-first ticket creation experience that feels like messaging rather than form-filling. The epic enables users to describe issues naturally, engage in threaded conversations with agents, attach files, and track ticket status.

**Epic Goal:** Make EasyPing usable for basic support workflows with natural conversation flows (AI enhancements come in Epic 3).

## Story

**As an** agent,
**I want** to reply to pings in a threaded conversation,
**so that** I can communicate back-and-forth with users naturally.

## Acceptance Criteria

1. Agent inbox shows all pings assigned to them or unassigned
2. Clicking ping opens conversation thread in right panel (split view)
3. Agent can type reply in message input box at bottom of thread
4. Agent message appears in thread immediately after sending
5. User receives realtime update (WebSocket via Supabase Realtime) - message appears without refresh
6. Conversation thread displays: sender avatar, name, message content, timestamp
7. Agent and user messages visually distinguished (e.g., different background colors)
8. Replying indicators show when agent or user is composing message

### Task 4: Implement Realtime Message Updates with Supabase Realtime (AC: 5)

**PURPOSE:** Enable realtime updates so users see new messages instantly without refreshing.

- [ ] Update `apps/web/src/components/inbox/inbox-client.tsx`:
  - [ ] Add Supabase Realtime subscription:
    ```typescript
    useEffect(() => {
      // Subscribe to new messages for this ping
      const channel = supabase
        .channel(`ping-messages:${ping.id}`)
        .on(
          'postgres_changes',
          {
            event: 'INSERT',
            schema: 'public',
            table: 'ping_messages',
            filter: `ping_id=eq.${ping.id}`,
          },
          async (payload) => {
            // Fetch full message with sender details
            const { data: newMessage } = await supabase
              .from('ping_messages')
              .select('*, sender:users(id, full_name, avatar_url, role)')
              .eq('id', payload.new.id)
              .single();

            if (newMessage) {
              // Only add if not already in state (avoid duplicate from optimistic update)
              setPing((prev) => {
                const exists = prev.messages.some((m) => m.id === newMessage.id);
                if (exists) return prev;

                return {
                  ...prev,
                  messages: [...prev.messages, newMessage],
                };
              });

              // Scroll to bottom
              scrollToBottom();

              // Show notification if message is from other user
              if (newMessage.sender_id !== currentUser.id) {
                toast.info(`New message from ${newMessage.sender.full_name}`);
              }
            }
          }
        )
        .subscribe();

      // Cleanup subscription on unmount
      return () => {
        supabase.removeChannel(channel);
      };
    }, [ping.id, currentUser.id]);
    ```

**API Patterns** [Source: docs/architecture/api-specification.md]

Next.js API route pattern for message sending:
```typescript
// apps/web/src/app/api/pings/[pingNumber]/messages/route.ts
import { createClient } from '@/lib/supabase/server';
import { NextRequest, NextResponse } from 'next/server';
import { z } from 'zod';

export async function POST(
  request: NextRequest,
  { params }: { params: { pingNumber: string } }
) {
  const supabase = await createClient();

  // Authenticate
  const { data: { user }, error: authError } = await supabase.auth.getUser();
  if (authError || !user) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
  }

  // Validate request
  const body = await request.json();
  const validation = sendMessageSchema.safeParse(body);
  if (!validation.success) {
    return NextResponse.json({ error: validation.error.issues }, { status: 400 });
  }

  // Fetch ping by ping_number
  const { data: ping } = await supabase
    .from('pings')
    .select('id, tenant_id')
    .eq('ping_number', parseInt(params.pingNumber))
    .single();

  if (!ping) {
    return NextResponse.json({ error: 'Ping not found' }, { status: 404 });
  }

  // Create message
  const { data: message, error } = await supabase
    .from('ping_messages')
    .insert({
      ping_id: ping.id,
      sender_id: user.id,
      content: validation.data.content,
      message_type: 'agent', // Determine based on user role
    })
    .select('*, sender:users(id, full_name, avatar_url, role)')
    .single();

  if (error) {
    return NextResponse.json({ error: 'Failed to send message' }, { status: 500 });
  }

  return NextResponse.json({ message }, { status: 201 });
}
```

**Supabase Realtime** [Source: docs/architecture/tech-stack.md]

Supabase Realtime provides WebSocket-based realtime updates for database changes and presence tracking:

1. **Database Changes (postgres_changes):**
   - Subscribe to INSERT events on `ping_messages` table
   - Filter by `ping_id` to only receive messages for current ping
   - Automatically updates UI when new message inserted

2. **Presence Tracking:**
   - Use Realtime Presence API to track who is replying
   - Each client broadcasts their replying state
   - Other clients receive presence sync events
   - Replying state auto-expires after 2 seconds of inactivity

Example Realtime subscription:
```typescript
const channel = supabase
  .channel('ping-messages:123')
  .on(
    'postgres_changes',
    {
      event: 'INSERT',
      schema: 'public',
      table: 'ping_messages',
      filter: 'ping_id=eq.123',
    },
    (payload) => {
      // Handle new message
    }
  )
  .subscribe();
```

### Debug Log References

**Major RLS Policy Issues Resolved:**

During implementation, several critical Row Level Security (RLS) policy issues were encountered that blocked realtime message functionality. These issues required creating three database migrations and implementing a split-query pattern to work around RLS restrictions.

**Issue 1: User Profile Query Blocked (Chicken-and-Egg Problem)**
- **Symptom**: "User not found" error when agent sends message
- **Root Cause**: API route couldn't query users table because RLS policy required `app.tenant_id` context, which browser clients don't set
- **Solution**: Created migration `20251111190409_allow_users_select_own_profile.sql` to allow users to query their own profile using `auth.uid()`

**Issue 2: Null Sender in Message Responses**
- **Symptom**: Runtime errors "Cannot read properties of null (reading 'id')" and "Cannot read properties of null (reading 'full_name')"
- **Root Cause**: Message INSERT with join failed due to RLS blocking the users table join
- **Solution**: Modified API to fetch user profile separately and manually construct `messageWithSender` object instead of using `.select('*, sender:users(...)')`

**Issue 3: Realtime Messages Not Appearing**
- **Symptom**: Agent could send message but it wouldn't appear in UI; later user messages wouldn't appear for agent
- **Root Cause**: Realtime subscription callbacks tried to join users table, which RLS blocked
- **Solution**:
  - Created migration `20251111191932_enable_realtime_for_ping_messages.sql` to enable Realtime publication
  - Implemented split-query pattern: fetch message first, then fetch sender separately by ID
  - Applied pattern to both inbox-client.tsx and ping-detail.tsx

**Issue 4: Cross-Component Sender Query Failure**
- **Symptom**: Agent inbox could receive messages, but user ping detail couldn't query sender profiles
- **Root Cause**: `users_select_own_profile` policy only allowed users to query their own profile, not other users in organization
- **Solution**: Created migration `20251111194434_allow_users_select_by_id.sql` with `USING (true)` to allow all authenticated users to SELECT user profiles by ID (read-only, safe for Realtime subscriptions)

**Issue 5: Database Reset Wiping Test Data**
- **Symptom**: Test data kept getting deleted when applying migrations
- **Root Cause**: Using `npx supabase db reset` instead of `npx supabase db push --local`
- **Solution**: Used `db push` for non-destructive migration application

**Critical Technical Pattern - Split-Query for RLS:**

To work around RLS policies blocking joins in Realtime subscriptions, we implemented a split-query pattern:

```typescript
// DON'T: This fails due to RLS blocking the join
const { data: newMessage } = await supabase
  .from('ping_messages')
  .select('*, sender:users(id, full_name, avatar_url, role)')
  .eq('id', payload.new.id)
  .single();

// DO: Query message and sender separately
const { data: newMessage } = await supabase
  .from('ping_messages')
  .select('*')
  .eq('id', payload.new.id)
  .single();

const { data: sender } = await supabase
  .from('users')
  .select('id, full_name, avatar_url, role')
  .eq('id', newMessage.sender_id)
  .single();

const transformedMessage = {
  ...newMessage,
  sender: sender as Pick<User, 'id' | 'full_name' | 'avatar_url' | 'role'>,
};
```
