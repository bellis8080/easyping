## Story

**As an** agent,
**I want** ping status to update automatically based on conversation flow (with manual override),
**so that** I can track ping lifecycle without manual status changes.

## Acceptance Criteria

### Auto-Status Transitions:

1. **When agent sends first message on New ping:** Status automatically changes New → In Progress
   - Records `first_response_at` timestamp (stops "First Response Time" SLA timer)
   - Auto-assigns ping to that agent if unassigned (see Story 2.5)
   - System message: "Agent Name started working on this ping"

2. **When user replies and status is Waiting on User:** Status automatically changes Waiting on User → In Progress
   - Indicates ball is back in agent's court (user provided requested information)
   - Resumes SLA resolution timer
   - System message: "User Name responded"

### Manual Status Changes (Agent-controlled):

3. Ping detail view shows status dropdown in toolbar (current status visible)
4. Agent can manually change status via dropdown (End users cannot change status)
5. Agent can manually set status to **Waiting on User** (e.g., when asking user to test a fix or provide info)
   - Pauses SLA resolution timer until user responds
   - System message: "Agent Name is waiting for User Name to respond"
6. Agent must manually mark pings as **Resolved** or **Closed** (not automated)
7. Manual status change creates system message in thread: "Agent changed status to Resolved"
8. Status change updates ping in database with timestamp

### Status Display & Filtering:

9. Ping list filters available: All, Active (combines New/In Progress/Waiting on User), Resolved, Closed
10. Status badge displayed on ping in conversation list (existing StatusIndicator component with icons and animations)
11. Resolved pings auto-archive from "My Pings" after 90 days (configurable in settings)
12. Closed pings hidden from default views (accessible via "Closed" filter tab)

### Task 4: Create Status Transition Logic Functions (AC: 1, 2, 5, 6)

**PURPOSE:** Encapsulate status transition business rules in reusable functions.

- [ ] Create `apps/web/src/lib/ping-status-transitions.ts`:
  ```typescript
  import { PingStatus, MessageType } from '@easyping/types';

  export interface StatusTransitionResult {
    newStatus: PingStatus;
    shouldCreateSystemMessage: boolean;
    systemMessageContent: string | null;
    timestampUpdates: Record<string, string>;
    shouldAutoAssign: boolean;
  }

  /**
   * Determines if and how ping status should change when a message is sent.
   * Returns null if no status change needed.
   */
  export function calculateStatusTransition(
    currentStatus: PingStatus,
    messageType: MessageType,
    senderId: string,
    pingCreatedBy: string,
    assignedTo: string | null,
    agentName: string,
    userName: string,
    firstResponseAt: string | null
  ): StatusTransitionResult | null {
    // Auto-transition: Agent replies to New ping → In Progress
    if (currentStatus === 'new' && messageType === 'agent' && !firstResponseAt) {
      return {
        newStatus: 'in_progress',
        shouldCreateSystemMessage: true,
        systemMessageContent: `${agentName} started working on this ping`,
        timestampUpdates: {
          first_response_at: new Date().toISOString(),
          last_agent_reply_at: new Date().toISOString(),
          status_changed_at: new Date().toISOString(),
        },
        shouldAutoAssign: !assignedTo, // Auto-assign if unassigned
      };
    }

    // Auto-transition: User replies to Waiting on User ping → In Progress
    if (currentStatus === 'waiting_on_user' && messageType === 'user' && senderId === pingCreatedBy) {
      return {
        newStatus: 'in_progress',
        shouldCreateSystemMessage: true,
        systemMessageContent: `${userName} responded`,
        timestampUpdates: {
          last_user_reply_at: new Date().toISOString(),
          status_changed_at: new Date().toISOString(),
        },
        shouldAutoAssign: false,
      };
    }

    // Track last reply timestamps without changing status
    if (messageType === 'agent') {
      return {
        newStatus: currentStatus,
        shouldCreateSystemMessage: false,
        systemMessageContent: null,
        timestampUpdates: {
          last_agent_reply_at: new Date().toISOString(),
        },
        shouldAutoAssign: false,
      };
    }

    if (messageType === 'user' && senderId === pingCreatedBy) {
      return {
        newStatus: currentStatus,
        shouldCreateSystemMessage: false,
        systemMessageContent: null,
        timestampUpdates: {
          last_user_reply_at: new Date().toISOString(),
        },
        shouldAutoAssign: false,
      };
    }

    return null;
  }

  /**
   * Validates manual status change and generates system message.
   */
  export function validateManualStatusChange(
    currentStatus: PingStatus,
    newStatus: PingStatus,
    agentName: string,
    userName: string
  ): { isValid: boolean; systemMessageContent: string | null; error: string | null } {
    // Prevent invalid transitions
    if (currentStatus === newStatus) {
      return { isValid: false, systemMessageContent: null, error: 'Status is already ' + newStatus };
    }

    // Generate appropriate system message
    let systemMessageContent = `${agentName} changed status to ${getStatusLabel(newStatus)}`;

    if (newStatus === 'waiting_on_user') {
      systemMessageContent = `${agentName} is waiting for ${userName} to respond`;
    }

    return { isValid: true, systemMessageContent, error: null };
  }

  function getStatusLabel(status: PingStatus): string {
    const labels: Record<PingStatus, string> = {
      new: 'New',
      in_progress: 'In Progress',
      waiting_on_user: 'Waiting on User',
      resolved: 'Resolved',
      closed: 'Closed',
    };
    return labels[status];
  }
  ```

### Task 6: Create Status Change API Endpoint (AC: 3, 4, 5, 6, 7, 8)

**PURPOSE:** Allow agents to manually change ping status via the existing dropdown UI.

- [ ] Create `apps/web/src/app/api/pings/[pingNumber]/status/route.ts`:
  ```typescript
  import { NextRequest, NextResponse } from 'next/server';
  import { createClient } from '@/lib/supabase/server';
  import { createAdminClient } from '@/lib/supabase/admin';
  import { z } from 'zod';
  import { validateManualStatusChange } from '@/lib/ping-status-transitions';
  import { getStatusLabel } from '@/lib/ping-status-utils';

  const updateStatusSchema = z.object({
    status: z.enum(['new', 'in_progress', 'waiting_on_user', 'resolved', 'closed']),
  });

  export async function PATCH(
    req: NextRequest,
    { params }: { params: { pingNumber: string } }
  ) {
    try {
      // Authenticate user
      const supabase = await createClient();
      const { data: { user }, error: authError } = await supabase.auth.getUser();

      if (authError || !user) {
        return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
      }

      // Get user profile and verify role
      const { data: userProfile } = await supabase
        .from('users')
        .select('id, full_name, role, tenant_id')
        .eq('id', user.id)
        .single();

      // Only agents/managers/owners can change status
      if (userProfile.role === 'end_user') {
        return NextResponse.json(
          { error: 'Forbidden: Only agents can change ping status' },
          { status: 403 }
        );
      }

      // Validate request body
      const body = await req.json();
      const validationResult = updateStatusSchema.safeParse(body);

      if (!validationResult.success) {
        return NextResponse.json(
          { error: 'Invalid request', details: validationResult.error },
          { status: 400 }
        );
      }

      const { status: newStatus } = validationResult.data;

      // Fetch ping
      const supabaseAdmin = createAdminClient();
      const { data: ping, error: pingError } = await supabaseAdmin
        .from('pings')
        .select('*, created_by_user:users!pings_created_by_fkey(full_name)')
        .eq('ping_number', parseInt(params.pingNumber))
        .eq('tenant_id', userProfile.tenant_id)
        .single();

      if (pingError || !ping) {
        return NextResponse.json({ error: 'Ping not found' }, { status: 404 });
      }

      // Validate status transition
      const validation = validateManualStatusChange(
        ping.status,
        newStatus,
        userProfile.full_name,
        ping.created_by_user.full_name
      );

      if (!validation.isValid) {
        return NextResponse.json({ error: validation.error }, { status: 400 });
      }

      // Update ping status
      const timestampUpdates: Record<string, string | null> = {
        status_changed_at: new Date().toISOString(),
      };

      if (newStatus === 'resolved' && !ping.resolved_at) {
        timestampUpdates.resolved_at = new Date().toISOString();
      }

      if (newStatus === 'closed' && !ping.closed_at) {
        timestampUpdates.closed_at = new Date().toISOString();
      }

      const { data: updatedPing, error: updateError } = await supabaseAdmin
        .from('pings')
        .update({
          status: newStatus,
          ...timestampUpdates,
        })
        .eq('id', ping.id)
        .select('*')
        .single();

      if (updateError) {
        console.error('Error updating ping status:', updateError);
        return NextResponse.json({ error: 'Failed to update status' }, { status: 500 });
      }

      // Create system message
      await supabaseAdmin.from('ping_messages').insert({
        ping_id: ping.id,
        sender_id: user.id,
        content: validation.systemMessageContent,
        message_type: 'system',
      });

      return NextResponse.json({ ping: updatedPing }, { status: 200 });
    } catch (error) {
      console.error('Error in status update:', error);
      return NextResponse.json({ error: 'Internal server error' }, { status: 500 });
    }
  }
  ```

### Status Transition State Machine

**Valid Transitions:**

```
New → In Progress (automatic when agent first replies)
New → Waiting on User (manual)
New → Resolved (manual)
New → Closed (manual)

In Progress → Waiting on User (manual)
In Progress → Resolved (manual)
In Progress → Closed (manual)

Waiting on User → In Progress (automatic when user replies)
Waiting on User → Resolved (manual)
Waiting on User → Closed (manual)

Resolved → In Progress (manual - if issue reopened)
Resolved → Closed (manual)

Closed → In Progress (manual - if ticket reopened)
```

**Automatic Transitions (Triggered by Message Send):**
1. **New → In Progress:** When agent sends first message on New ping
   - Auto-assigns ping to that agent (if unassigned)
   - Records `first_response_at` timestamp
   - Creates system message: "Agent Name started working on this ping"

2. **Waiting on User → In Progress:** When user replies to Waiting on User ping
   - Resumes SLA resolution timer
   - Creates system message: "User Name responded"

**Manual Transitions (Agent-controlled via dropdown):**
- All other status changes are manual
- Agent can set any valid status at any time
- Status change creates system message with agent name

### SLA Timer Logic (Future Story 5.1)

Status transitions affect SLA timers:
- **First Response Timer:** Stops when `first_response_at` is set (agent sends first reply)
- **Resolution Timer:** Pauses when status = Waiting on User, resumes when status = In Progress
- **Resolution Timer:** Stops when status = Resolved or Closed

### System Message Format

System messages provide conversation context and audit trail:

```typescript
// Examples:

"John Smith responded"
"Sarah Chen changed status to Resolved"

```

**Implementation:**

- `sender_id` = Agent who triggered the action (for audit trail)
- `content` = Human-readable message with user names
- Displayed in thread with distinct styling (gray background, italicized, centered)

### Auto-Assignment Logic (AC: 1)

When agent sends first reply to New ping:
1. Check if `assigned_to` is NULL
2. If NULL, set `assigned_to = agent.id`
3. Create system message: "Agent Name started working on this ping"

This ensures pings are automatically assigned when agents engage, reducing manual assignment overhead.

### Real-Time Updates (AC: 8)

Status changes must propagate in real-time:
1. Agent A changes status via dropdown
2. API updates `pings` table with new status
3. Supabase Realtime broadcasts UPDATE event
4. Agent B viewing same ping receives event and updates UI
5. System message appears in conversation thread for both agents

**Realtime Subscription Pattern:**
```typescript
channel
  .on('postgres_changes', {
    event: 'UPDATE',
    schema: 'public',
    table: 'pings',
    filter: `id=eq.${ping.id}`,
  }, (payload) => {
    setSelectedPing(prev => prev ? { ...prev, status: payload.new.status } : null);
  })
  .subscribe();
```

**Issue 4: Real-time subscriptions breaking with infinite loop**
- **Symptom**: Messages sent but not appearing, real-time subscriptions recreating infinitely
