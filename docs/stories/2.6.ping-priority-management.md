## Story

**As an** agent,
**I want** to set ping priority (Low, Normal, High, Urgent),
**so that** I can triage and focus on critical issues first.

## Acceptance Criteria

1. Ping toolbar includes priority dropdown (Low, Normal, High, Urgent)
2. Priority change updates database and creates system message
3. Default priority: Normal (set on ping creation)
4. Urgent pings highlighted in ping list (red indicator)
5. Ping list sortable by priority
6. Priority badge displayed in ping conversation list

    ```typescript
    .order('priority', {
      ascending: false, // urgent first, then high, normal, low
      nullsFirst: false
    })
    ```
  - Note: PostgreSQL sorts text alphabetically by default (high < low < normal < urgent)
  - To sort correctly (urgent > high > normal > low), use CASE expression:
    ```sql
    ORDER BY CASE priority
      WHEN 'urgent' THEN 1
      WHEN 'high' THEN 2
      WHEN 'normal' THEN 3
      WHEN 'low' THEN 4
    END ASC
    ```

### Database Schema

**Pings Table** [Source: docs/architecture/database-schema.md#pings-table]
- Table: `pings`
- Priority column: `priority TEXT NOT NULL DEFAULT 'normal'`
- CHECK constraint: `pings_priority_valid CHECK (priority IN ('low', 'normal', 'high', 'urgent'))`
- Index: No specific index for priority (filtered queries will use `idx_pings_tenant_id`)
- RLS enabled: `tenant_id = current_setting('app.tenant_id', true)::UUID`

**Ping Messages Table** [Source: docs/architecture/database-schema.md#ping-messages-table]
- Used for system messages when priority changes
- `message_type` must be 'system'
- `content` contains the priority change message

### API Endpoint Specifications

**PATCH /api/pings/[pingNumber]/priority** [Based on Story 2.5 patterns]
- Request body: `{ priority: 'low' | 'normal' | 'high' | 'urgent' }`
- Response: `{ ping: Ping }` (updated ping object)
- Status codes:
  - 200: Success
  - 400: Invalid request (bad priority value, no change, validation error)
  - 401: Unauthorized (not authenticated)
  - 403: Forbidden (end user attempting to update)
  - 404: Ping not found (or tenant mismatch)
  - 500: Database error

### Interaction Design Patterns

**Priority Change Flow** [User Experience]
1. Agent opens ping detail view
2. Clicks priority dropdown in toolbar
3. Dropdown opens with 4 options (Low, Normal, High, Urgent)
4. Agent selects new priority (e.g., "Urgent")
5. **Immediate Feedback:**
   - Dropdown closes
   - Badge updates instantly (optimistic UI)
   - Loading spinner appears briefly in dropdown trigger
6. **API Request:**
   - PATCH request sent to `/api/pings/[pingNumber]/priority`
   - If success: Toast notification "Priority updated to Urgent"
   - If error: Badge reverts, error toast shown
7. **System Message:**
   - New message appears in thread: "Priority changed to urgent"
   - Message attributed to agent who made change
8. **Real-Time Propagation:**
   - Other agents viewing same ping see badge update
   - Ping list re-sorts if sorted by priority

**Visual Feedback States**
- **Idle:** Dropdown shows current priority with icon
- **Hover:** Dropdown trigger highlights (bg-gray-100)
- **Open:** Dropdown menu overlays toolbar, selected option checked
- **Loading:** Spinner replaces dropdown icon, dropdown disabled
- **Success:** Brief green pulse on badge, toast notification
- **Error:** Red shake animation on dropdown, badge reverts, error toast
