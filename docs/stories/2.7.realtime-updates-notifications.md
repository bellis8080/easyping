# Story 2.7: Real-Time Updates & Notifications

## Epic Context

This story is part of **Epic 2: Chat-First Ping Creation & Threading**, which delivers EasyPing's core differentiator - a conversational, chat-first ticket creation experience that feels like messaging rather than form-filling. The epic enables users to describe issues naturally, engage in threaded conversations with agents, attach files, and track ticket status and priority.

**Epic Goal:** Make EasyPing usable for basic support workflows with natural conversation flows (AI enhancements come in Epic 3).

## Status

Done

## Story

**As a** user or agent,
**I want** to see new messages and updates instantly without refreshing,
**so that** conversations feel live and responsive.

## Acceptance Criteria

1. Supabase Realtime subscriptions configured for ping messages and status changes
2. New messages appear in conversation thread within 500ms (no page refresh)
3. Ping list updates when new pings created or status changes
4. In-app notification badge shows unread ping count
5. Browser tab title updates with unread count: "(3) EasyPing"
6. Sound notification plays when new message received (user can disable in settings)
7. Connection status indicator shows when realtime connection lost

### Task 2: Implement Unread Message Tracking (AC: 4, 5)

**PURPOSE:** Track which pings have unread messages and display unread counts to users.

- [ ] Create `ping_reads` database table (migration required)
  - Schema:
    ```sql
    CREATE TABLE ping_reads (
      id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
      ping_id UUID NOT NULL REFERENCES pings(id) ON DELETE CASCADE,
      user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
      last_read_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
      last_read_message_id UUID REFERENCES ping_messages(id) ON DELETE SET NULL,
      created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
      updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
      UNIQUE(ping_id, user_id)
    );
    CREATE INDEX idx_ping_reads_user_id ON ping_reads(user_id);
    CREATE INDEX idx_ping_reads_ping_id ON ping_reads(ping_id);
    ```
  - Purpose: Track when each user last read each ping
  - Unique constraint: One read record per user per ping
  - Indexes: Optimize queries by user_id and ping_id
- [ ] Create API endpoint to mark ping as read
  - Route: `POST /api/pings/[pingNumber]/read`
  - Request body: `{ lastReadMessageId: string }` (UUID of last message viewed)
  - Authentication: Verify user is authenticated
  - Upsert `ping_reads` record:
    ```typescript
    await supabase.from('ping_reads').upsert({
      ping_id: ping.id,
      user_id: user.id,
      last_read_at: new Date().toISOString(),
      last_read_message_id: lastReadMessageId,
      updated_at: new Date().toISOString()
    }, { onConflict: 'ping_id,user_id' });
    ```
  - Return success status
- [ ] Calculate unread count for each ping in inbox query
  - Modify inbox page server component query to join `ping_reads` table
  - Calculate unread count using subquery:
    ```sql
    SELECT p.*,
      (SELECT COUNT(*) FROM ping_messages pm
       WHERE pm.ping_id = p.id
       AND pm.created_at > COALESCE(pr.last_read_at, '1970-01-01')
       AND pm.sender_id != current_user_id) AS unread_count
    FROM pings p
    LEFT JOIN ping_reads pr ON pr.ping_id = p.id AND pr.user_id = current_user_id
    ```
  - Include `unread_count` in `PingWithRelations` type
- [ ] Display unread badge on ping list items
  - Location: `inbox-client.tsx` ping list rendering
  - Show badge if `ping.unread_count > 0`
  - Badge design: Blue circle with white number, positioned top-right of ping item
  - Badge component from shadcn/ui:
    ```tsx
    {ping.unread_count > 0 && (
      <Badge variant="default" className="ml-auto bg-blue-500 text-white">
        {ping.unread_count}
      </Badge>
    )}
    ```
- [ ] Mark ping as read when user opens conversation
  - Call `POST /api/pings/[pingNumber]/read` when user selects ping
  - Pass ID of most recent message as `lastReadMessageId`
  - Optimistically update local state (clear unread badge immediately)
  - On success: Update `ping_reads` state
  - On error: Revert optimistic update
- [ ] Calculate total unread count across all pings
  - Sum all `ping.unread_count` values in inbox

  - Update when pings array changes or new messages arrive

### Task 6: Optimize Real-Time Performance (AC: 2)

**PURPOSE:** Ensure real-time updates are fast (<500ms) and don't cause UI jank or performance issues.

- [ ] Implement optimistic UI updates

  - Same pattern for status changes, assignments, priority updates
  - Reconcile with real-time event when it arrives (update IDs, timestamps)
- [ ] Debounce real-time update handlers
  - Prevent rapid successive updates from causing multiple re-renders

  - Debounce threshold: 100ms for message updates, 300ms for ping list updates
  - Balance: Fast enough to feel instant, slow enough to batch updates
- [ ] Optimize React re-renders
  - Use `React.memo()` for ping list items to prevent unnecessary re-renders
  - Use `useMemo()` for expensive calculations (unread counts, filtered lists)
  - Use `useCallback()` for event handlers passed as props
  - Measure: Use React DevTools Profiler to identify slow components
- [ ] Implement virtual scrolling for long ping lists
  - Use `react-virtual` or `react-window` library
  - Render only visible ping items (e.g., 20 at a time)
  - Improves performance when inbox has 100+ pings
  - Maintain scroll position when new pings arrive
- [ ] Test performance with high message volume
  - Simulate 10 messages arriving within 1 second

  - Measure time from message insert to UI update (should be <500ms)
  - Use Performance API to measure render times

### Supabase Realtime Patterns

**Subscription Setup** [Source: Supabase docs + existing implementation]
```typescript
const channel = supabase
  .channel('inbox-realtime')
  .on(
    'postgres_changes',
    {
      event: 'INSERT',
      schema: 'public',
      table: 'ping_messages',
      filter: `ping_id=eq.${pingId}`,
    },
    (payload) => {
      // Handle new message
      const newMessage = payload.new as PingMessage;
      setPings((prev) => /* update pings with new message */);

      // Play notification sound if from another user
      if (newMessage.sender_id !== currentUser.id && soundEnabled) {
        playNotificationSound();
      }
    }
  )
  .on(
    'postgres_changes',
    {
      event: 'UPDATE',
      schema: 'public',
      table: 'pings',
      filter: `tenant_id=eq.${tenantId}`,
    },
    (payload) => {
      // Handle ping update (status, priority, assignment)
      const updatedPing = payload.new as Ping;
      setPings((prev) => /* update ping in list */);
    }
  )
  .on('system', { event: 'SUBSCRIBED' }, () => {
    setIsConnected(true);
  })
  .on('system', { event: 'CHANNEL_ERROR' }, () => {
    setIsConnected(false);
    // Trigger reconnection logic
  })
  .subscribe();

// Cleanup on unmount
return () => {
  supabase.removeChannel(channel);
};
```

**Performance Optimization**
- Use `filter` parameter to reduce payload size (only subscribe to relevant events)
- Use `select` parameter to specify exact fields needed (avoid fetching entire rows)
- Debounce event handlers to batch rapid updates
- Implement optimistic UI updates (don't wait for real-time event)

## Known Issues

### Inbox Badge Count Race Condition (Minor)

**Issue:** When an agent is viewing a ping and receives a new message in that same ping, the inbox badge count sometimes increments by 1 (showing the message as unread) even though the message is automatically marked as read. The badge count corrects itself after a page refresh.

**Expected Behavior:** Badge count should remain at 0 when viewing the ping where the message arrives (Scenario 1).

**Actual Behavior:** Badge count increments by 1 temporarily until page refresh.

**Root Cause:** Race condition between:
1. Mark-as-read operation (writes to `ping_reads` table)
2. Badge count recalculation (queries `ping_reads` and `ping_messages` tables)
3. Polling interval (triggers recalculation every 5 seconds)

The critical window mechanism was implemented to prevent polling from interfering with the delayed recalculation, but the race condition still occurs occasionally.

**Impact:** Low - visual inconsistency only, corrects on refresh, does not affect data integrity

**Workaround:** Refresh the page to see correct badge count

**Status:** Documented as known issue, to be addressed in future performance optimization story
