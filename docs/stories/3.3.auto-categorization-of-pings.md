## Story

**As an** end user,
**I want** Echo (AI assistant) to understand my problem through natural conversation and confirm understanding before sending to an agent,
**so that** agents receive clear, well-defined issues without needing to ask basic clarifying questions.

**As an** agent,
**I want** pings to arrive with AI-generated problem summaries, titles, and categories,
**so that** I can immediately understand the issue and start working on solutions.

## Acceptance Criteria

1. Default categories created on setup: Hardware, Software, Network, Access Request, Password Reset, Other, Needs Review
2. **During ping creation flow, Echo engages user in conversation to understand the problem BEFORE making ping visible to agents**
3. **Echo introduces himself on first encounter:**
   - "Hi! I'm Echo, your AI assistant. I'll ask a few quick questions to understand your issue, then connect you to the right person."
4. **Echo asks clarifying questions focused on understanding the PROBLEM (not categorizing):**
   - Questions like: "What exactly is happening?", "When did this start?", "What error message do you see?"
   - NOT category-focused questions like: "Is this hardware or software?"
   - Echo maintains conversation context across all questions
   - Echo continues until problem is well-defined
5. **Once Echo understands the problem, Echo generates a problem statement and confirms with user:**
   - "Let me make sure I understand: [problem statement]. Is that correct?"
   - "Once you confirm, I'll send this to an agent for resolution."
6. **User can confirm or provide corrections:**
   - If user confirms: Proceed to finalization
   - If user disagrees or adds detail: Echo revises problem statement and confirms again
   - Allow multiple confirmation attempts (no hard limit)
7. **After confirmation, Echo finalizes the ping:**
   - Determine category from confirmed problem statement
   - Generate title from problem statement
   - Change ping `status` from `'draft'` to `'new'` (becomes visible in agent inbox)
   - AI-generated problem statement stored as `ai_summary` (pinned at top of ping)
   - Send confirmation to user: "Thanks! I've categorized this as [Category] and sent it to our team. An agent will be with you shortly."
8. **Soft limit of ~10 clarifying questions:**
   - Not a hard cutoff
   - Echo determines when it's not making progress with user
   - If stuck: Finalize with "Other" category and best-effort problem statement
   - Let AI be intelligent about when human help is needed
9. **If AI is not configured (no API key), gracefully skip Echo conversation:**
   - Ping created with `status = 'new'` immediately (bypass draft status)
   - User prompted to enter title manually (simple text input)
   - Category defaults to "Needs Review"
   - No `ai_summary` generated (field remains null)
   - Ping appears in agent inbox immediately for manual triage
   - UI shows notice: "AI assistant unavailable. Your ping has been sent to our team for review."
10. **Problem statement summary pinned at top of ping detail view** (collapsible card, above conversation thread)
11. **All conversation messages (user + Echo questions + confirmation) stored as initial ping thread entries**
12. Category displayed as badge in ping list and ping detail view
13. Agents can manually override AI category selection (dropdown in toolbar showing all active categories)
14. System message created when finalized: "Echo categorized as: Software (92% confidence)"
15. Category statistics shown in analytics dashboard (coming in Epic 5)

### Task 5: Create Echo Conversation Service (AC: 4, 5, 6, 7, 8)

**PURPOSE:** Implement service that guides conversation to understand the problem, not just categorize.

- [ ] Create file: `apps/web/src/lib/services/echo-conversation-service.ts`
- [ ] Implement `analyzeConversation(conversation: string[]): Promise<ConversationAnalysis>`:
  - Use AI to analyze conversation for problem clarity
  - Return: `{ problemUnderstood: boolean, nextQuestion?: string, problemStatement?: string, confidence: number }`
  - **Focus on problem understanding, NOT categorization:**
    - Check if core issue is clear (what's happening, when it started, what user tried)
    - Identify gaps in understanding
    - Generate next question to fill gaps
  - If `problemUnderstood: true`: Generate problem statement for confirmation
  - If `problemUnderstood: false`: Generate next clarifying question
- [ ] Implement `generateNextQuestion(conversation: string[], missingInfo: string[]): Promise<string>`:
  - Use AI to generate problem-focused question (NOT category-focused)
  - Examples of GOOD questions:
    - "What exactly is happening when you try to [action]?"
    - "When did this start?"
    - "What error message do you see?"
    - "Have you tried anything to fix this?"
  - Examples of BAD questions (too category-focused):
    - "Is this a hardware or software issue?"
    - "Which category does this belong to?"
  - Return natural, conversational question
- [ ] Implement `generateProblemStatement(conversation: string[]): Promise<string>`:
  - Summarize entire conversation into 2-3 sentence problem statement
  - Format: "User is experiencing [specific issue]. [Key details: when, where, what]. [Attempted solutions or additional context]."
  - Example: "User unable to log into email since this morning. Outlook shows 'Credentials incorrect' error. User has tried resetting password but issue persists."
- [ ] Implement `determineWhenToEscalate(conversation: string[], clarificationCount: number): Promise<boolean>`:
  - Let AI decide if conversation is stuck (not making progress)
  - Factors to consider:
    - User responses are vague or unhelpful
    - Questions going in circles (asking same thing different ways)
    - Clarification count approaching 10
    - User explicitly asks for human help
  - Return `true` if should finalize with current info and escalate to agent
- [ ] Add 5-second timeout for each AI call (uses `Promise.race`)
- [ ] Add fallback logic:
  - If AI unavailable: Generate generic follow-up question or escalate to agent
  - If timeout: Use last known state or escalate

### Architecture Context

**CRITICAL DESIGN PRINCIPLE: Problem-First, Category-Second**

The core insight of this story is that categorization should be a BYPRODUCT of understanding the problem, not the goal itself.

**Wrong Approach (category-first):**
1. User: "My email isn't working"
2. Echo: "Is this a hardware or software issue?" ❌ (trying to categorize too early)
3. Echo: "Which application is it?" ❌ (still category-focused)
4. Assign category based on vague conversation
5. Generate problem statement as afterthought

**Correct Approach (problem-first):**
1. User: "My email isn't working"
2. Echo: "What exactly happens when you try to use email?" ✅ (understanding the problem)
3. User: "I can't log in"
4. Echo: "What error message do you see?" ✅ (getting specifics)
5. User: "It says credentials incorrect"
6. Echo: "When did this start?" ✅ (understanding timeline)
7. User: "This morning after I changed my password"
8. **NOW Echo understands the problem:**
   - Problem statement: "User unable to log into email since changing password this morning. Error: 'Credentials incorrect'."
9. Echo confirms: "Let me make sure I understand: You changed your password this morning and now can't log into email with error 'Credentials incorrect'. Is that correct?"
10. User: "Yes"
11. **ONLY NOW does categorization happen:**
    - Category: "Password Reset" (derived from problem)
    - Title: "Email login failed after password change"

**Complete Conversation Flow:**

```
1. USER SENDS FIRST MESSAGE
   ↓
2. PING CREATED (status = 'draft', title = null)
   ↓
3. ECHO INTRODUCTION (first encounter only)
   "Hi! I'm Echo, your AI assistant. I'll ask a few quick questions
    to understand your issue, then connect you to the right person."
   ↓
4. PROBLEM DISCOVERY PHASE
   Loop (soft limit ~10 questions):
   ├─ Echo asks clarifying question (problem-focused)
   ├─ User responds
   ├─ AI analyzes: "Do I understand the problem clearly?"
   ├─ If NO → Generate next question, continue loop
   └─ If YES → Generate problem statement, go to step 5

   If stuck (not making progress):
   └─ AI decides to escalate → Go to step 7 with "Other" category
   ↓
5. CONFIRMATION PHASE
   Echo: "Let me make sure I understand: [problem statement].
          Is that correct? Once you confirm, I'll send this
          to an agent for resolution."
   ↓
6. USER CONFIRMATION
   User confirms:
   └─ Go to step 7

   User disagrees/adds detail:
   └─ Revise problem statement
   └─ Go back to step 5 (reconfirm)
   ↓
7. FINALIZATION (after confirmation OR escalation)
   ├─ Categorize based on problem statement (not raw conversation)
   ├─ Generate title from problem statement
   ├─ Update ping: status = 'new', set title, ai_summary, category_id
   ├─ Send system message: "Echo categorized as: {category} ({confidence}%)"
   └─ Send user confirmation: "Thanks! I've categorized this as {category}
       and sent it to our team. An agent will be with you shortly."
   ↓
8. PING APPEARS IN AGENT INBOX
   Agents see:
   - Problem statement pinned at top
   - AI-generated title
   - Category badge
   - Full conversation history
```

**Status-Based Visibility:**
- `status = 'draft'` → Hidden from agent inbox, user can see/reply, Echo is conversing
- `status = 'new'` → Visible in agent inbox, categorized, has title/summary, ready for triage
- Agent inbox query: `WHERE status IN ('new', 'in_progress', 'waiting_on_user', 'resolved')`

**Soft 10-Question Limit:**
- Not a hard cutoff at exactly 10 questions
- AI determines when it's not making progress
- Factors AI considers:
  - User responses are vague, unhelpful, or repetitive
  - Questions going in circles (asking same thing different ways)
  - Approaching 10 questions without clarity
  - User explicitly asks for human help ("Can I just talk to a person?")
- When stuck: Finalize with "Other" category and best-effort problem statement

**Confirmation Step is Critical:**
- Gives user control and catches misunderstandings
- User can disagree/add detail before agents see it
- Reduces back-and-forth with agents later
- Improves accuracy of problem statement
- Builds trust in AI system

**Graceful Handling When AI Not Configured:**
- Check if organization has AI config: `ai_config.provider` and `ai_config.encrypted_api_key`
- **If NO AI configured:**
  - Skip Echo conversation entirely
  - Ping created with `status = 'new'` immediately (no draft phase)
  - User must provide title manually (required field in create ping form)
  - Category defaults to "Needs Review"
  - No `ai_summary` generated (field remains null)
  - UI shows notice: "AI assistant unavailable. Your ping has been sent to our team for review."
  - Ping appears in agent inbox immediately for manual triage
- **If AI configured:**
  - Normal Echo conversation flow (draft → questions → confirmation → finalization)
- This ensures EasyPing works even before owner configures AI providers

## Future Work

### Scope Awareness for Echo

**Issue Identified:** During testing, users can confuse Echo by sending completely off-topic messages (e.g., "I'm hungry", "order me pizza"). Echo currently has no way to recognize when requests are outside the organization's support scope and redirect users appropriately.

**Problem:** EasyPing is designed to be flexible - organizations may use it for IT support, HR inquiries, facilities management, general customer service, or any other support function. We cannot hardcode what topics are "in scope" vs "out of scope" because this varies by organization.

**Proposed Solution (Future Story):**

1. **Organization-Level Scope Configuration:**
   - Add a `support_scope` or `echo_context` field to the `organizations` table
   - Allow organization owners to describe what their support team handles
   - Example: "IT support for employees - computers, software, network, accounts"
   - Example: "HR inquiries - benefits, payroll, time off, policies"
   - Example: "General customer support for our e-commerce platform"

2. **Echo Prompt Enhancement:**
   - Include the organization's scope description in Echo's AI prompts
   - Echo can then recognize when requests fall outside the defined scope
   - Echo politely redirects users: "I'm specifically here to help with [organization's scope]. Is there something related to that I can assist with?"

3. **UI for Scope Configuration:**
   - Add to organization settings (same page as AI provider config)
   - Textarea for owners to describe their support scope
   - Optional: predefined templates (IT Support, HR, Customer Service, etc.)

**Acceptance Criteria for Future Story:**
- Organization owners can define what their support team handles
- Echo uses this context to determine if requests are in-scope
- Out-of-scope requests receive polite redirects (not processed as valid pings)
- Scope definition is optional (if not set, Echo accepts all requests)

**Note:** This was identified during UAT testing when a user entered "I'm hungry" and Echo attempted to create a problem statement about hunger. The current implementation proceeds with any request, which can lead to nonsensical pings being created.
