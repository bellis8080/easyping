## Story

**As an** agent,
**I want** pings automatically categorized (Hardware, Software, Network, Access Request, etc.) with AI-powered conversational clarification when ambiguous,
**so that** I can quickly understand ping type and route appropriately without manual triage.

## Acceptance Criteria

1. Default categories created on setup: Hardware, Software, Network, Access Request, Password Reset, Other
2. When ping created, AI analyzes first user message and assigns category from available categories
3. Category determination completes within 3 seconds
4. Category displayed as badge in ping list and ping detail view
5. Agents can manually override AI category selection (dropdown in toolbar showing all active categories)
6. System message created when category assigned: "AI categorized as: Software"
7. Fallback to "Other" or first available category if AI fails or confidence below threshold (70%)
8. Category statistics shown in analytics dashboard (coming in Epic 5)

**Additional AC from PRD Note:**
> The PRD story title says "Auto-Categorization of Tickets" but this should be **Auto-Categorization of Pings with Conversational Clarification**. The clarification feature is implicit in the epic goal of AI-native support - when the AI is uncertain about categorization (confidence < 70%), it should engage the user in a brief clarifying conversation before assigning a category. This prevents miscategorization and provides a better user experience.

9. **When AI confidence < 70%**, system generates a clarifying question and sends it as a system message to the user (e.g., "To help route your ping correctly, could you clarify: Is this a hardware issue or a software issue?")
10. User's response to clarifying question is analyzed to improve category confidence
11. After clarification, AI re-categorizes with improved context
12. Maximum 1 clarifying question per ping (prevents infinite loops)

### Task 3: Implement AI Categorization Service (AC: 2, 3, 7, 9, 10, 11)

**PURPOSE:** Create service layer that calls AI provider to categorize pings with clarification logic.

- [ ] Create file: `apps/web/src/lib/services/categorization-service.ts`
- [ ] Implement `categorizePing(pingId: UUID, conversation: string[], availableCategories: Category[]): Promise<CategoryAssignment>` function:
  - Fetch AI configuration from organization
  - Call AI provider's `categorize()` method with conversation and category list
  - Parse `CategoryResult` (category name, confidence, reasoning)
  - If confidence >= 0.70: Return category assignment
  - If confidence < 0.70: Generate clarifying question based on reasoning
  - Return `{ needsClarification: true, clarifyingQuestion: string, topCandidates: string[] }`
- [ ] Implement `recategorizePing(pingId: UUID, previousConversation: string[], clarificationResponse: string): Promise<CategoryAssignment>` function:
  - Append clarification response to conversation context
  - Re-run categorization with enhanced context
  - Force category assignment (use highest confidence result, even if < 70%)
  - Increment `clarification_attempts` counter
- [ ] Add error handling:
  - If AI provider unavailable, fallback to "Other" category
  - Log categorization failures to console
  - Set `category_confidence = 0` for fallback categorizations
- [ ] Add timeout: Categorization must complete within 3 seconds (AC: 3)
  - Use `Promise.race()` with 3000ms timeout
  - On timeout, fallback to "Other" category

### Task 4: Create API Route for Ping Categorization (AC: 2, 6, 9, 11)

**PURPOSE:** Expose categorization service via API endpoint callable during ping creation and after clarification.

- [ ] Create file: `apps/web/src/app/api/pings/[pingNumber]/categorize/route.ts`
- [ ] Implement `POST /api/pings/[pingNumber]/categorize` endpoint:
  - Extract `pingNumber` from URL params
  - Fetch ping and messages from database (with `tenant_id` filter)
  - Build conversation array from `ping_messages`
  - Fetch active categories for organization
  - Call `categorizePing()` service
  - **If categorization successful:**
    - Update `pings` table: `category_id`, `category_confidence`, `category_reasoning`
    - Create system message: "AI categorized as: {categoryName}" (with confidence in metadata)
    - Return `{ success: true, category: Category, confidence: number }`
  - **If needs clarification:**
    - Update `pings.needs_clarification = true`
    - Create system message with clarifying question
    - Return `{ success: true, needsClarification: true, question: string }`

  - Return 403 if user not authorized for ping (not creator and not agent)
  - Return 404 if ping not found
  - Return 500 on AI provider errors (with fallback to "Other")
- [ ] Implement `PUT /api/pings/[pingNumber]/categorize` endpoint (for manual override by agents):
  - Accept `{ categoryId: UUID }` in request body
  - Validate category exists and is active
  - Update `category_id` with manual override
  - Set `category_confidence = 1.0` (manual overrides are 100% confident)
  - Create system message: "{agentName} changed category to: {categoryName}"
  - Return updated ping

**Clarification Workflow Design** [Design Decision]
- **When to clarify:** AI confidence < 70% threshold
- **Clarification method:** System message with question (appears in conversation thread)
- **User response:** Next user message treated as clarification response
- **Re-categorization:** Automatically triggered when user replies to ping with `needs_clarification = true`
- **Maximum attempts:** 1 clarifying question per ping (prevents infinite loops)
- **Example flow:**
  1. User: "My computer isn't working"
  2. AI: Confidence 55% → Needs clarification
  3. System: "To help route your ping correctly, is this a hardware issue (physical device) or software issue (application/program)?"
  4. User: "It's the monitor, the screen is black"
  5. AI re-categorizes with enhanced context → "Hardware" (85% confidence)
  6. System: "AI categorized as: Hardware"
