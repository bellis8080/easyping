## Story

**As a** manager,
**I want** to organize agents into teams with dedicated inboxes in the sidebar,
**so that** pings can be routed to specialized teams and agents can focus on their team's work.

**As an** agent,
**I want** to see my team's pings in a dedicated inbox,
**so that** I can claim and work on pings relevant to my expertise.

## Design Philosophy

This story embraces EasyPing's **chat-first, conversational approach** by making teams a first-class navigation concept rather than a hidden settings configuration. Teams appear as **inboxes in the sidebar**, just like "Assigned to Me" or "Unassigned". This keeps agents focused on their team's work and provides immediate visibility into team workload.

**Key UX Principles:**
- Teams are navigation items, not settings
- Creating a team = creating a new inbox
- Team membership is managed inline in the inbox header (not buried in settings)
- Pings route to team inboxes but stay unassigned until an agent claims them
- Agents only see inboxes for teams they belong to
- Managers/owners see all team inboxes and can create new ones

## Acceptance Criteria

### Part A: Team Inboxes in Sidebar Navigation

1. Sidebar shows expandable "Team Inboxes" section after standard filters (All Pings, Assigned to Me, Unassigned, Urgent, Closed)
2. Each team appears as a navigation item with: Team name, unread ping count badge
3. Agents only see teams they are members of
4. Managers and owners see all teams in the organization
5. Clicking a team inbox filters the ping list to show only pings assigned to that team
6. "Create Team" button visible only to managers/owners, opens inline form or modal
7. Maximum 20 teams per organization

### Part B: Team Inbox View

8. When viewing a team inbox, the header shows the team name (not "Agent Inbox")
9. Header area shows team members as avatar pills/badges
10. Manager/owner can click team members area to manage membership (add/remove agents)
11. Team description shown in header (if set) as subtle subtitle
12. Pings in team inbox are filterable by status (New, In Progress, etc.) and sortable
13. Unassigned pings in team inbox show "Claim" button for agents to take ownership
14. When agent claims a ping, it remains in the team inbox but also appears in "Assigned to Me"

### Part C: Routing Rules

15. Routing rules configurable in Settings (manager/owner only): Map categories to teams or specific agents
16. Example rules: "Hardware → Hardware Team", "Password Reset → Bob"
17. When ping is categorized by Echo, routing rule is applied automatically
18. If rule routes to team: Ping assigned to team (appears in team inbox, unassigned to specific agent)
19. If rule routes to specific agent: Ping assigned directly to that agent
20. System message created: "Auto-routed to Hardware Team" or "Auto-assigned to Bob"
21. If no routing rule matches, ping remains unassigned (current behavior)
22. Routing executed immediately after Echo categorization (during ping creation flow)
23. Manual assignment (claiming or reassigning) always takes precedence over routing

### Part D: Team Management

24. Manager/owner can create teams from sidebar "Create Team" action
25. Team form: Name (required, unique), Description (optional)
26. Manager/owner can edit team name and description via team inbox header
27. Manager/owner can add/remove agents from team via inline member management
28. An agent can belong to multiple teams
29. Team cannot be deleted if it has members or pings (must reassign first)
30. Deleting a team moves its pings to unassigned

### Part E: Empty States

31. Team inbox with no pings: "No pings in {Team Name} yet. When pings are categorized and routed here, they'll appear in this inbox."
32. Agent not on any team: "You're not a member of any team yet. Ask your manager to add you to a team."
33. No teams in organization (manager view): "No teams created yet. Create your first team to organize your agents by specialty."
34. Member management with no available agents: "All agents are already members of this team."
35. Routing rules empty state: "No routing rules configured. Pings will remain unassigned until an agent claims them."

### Part F: Error States

36. Duplicate team name: Inline form error "A team with this name already exists"
37. Max teams reached: Disable "Create Team" with tooltip "Maximum 20 teams reached. Delete an existing team to create a new one."
38. Failed to add member: Toast error "Failed to add {Agent Name}. Please try again." with retry option
39. Ping already claimed: Toast "This ping was just claimed by {Agent Name}" and refresh ping list
40. Permission denied on team inbox: Redirect to main inbox with toast "You don't have access to this team"
41. Failed to claim ping: Toast error "Failed to claim ping. Please try again."

### Part G: Ping Team Reassignment (Escalation)

42. Ping detail toolbar shows current team assignment (if any) as a dropdown
43. Manager/owner can reassign any ping to a different team from the team dropdown
44. Agent can reassign a ping they own (assigned_to = self) to a different team
45. When ping is reassigned to a different team:
    - `team_id` updated to new team
    - `assigned_to` cleared (ping becomes unassigned in new team inbox)
    - System message created: "Reassigned from {Old Team} to {New Team}" (or "Assigned to {New Team}" if no previous team)
46. Agent can also reassign to "No Team" to remove team assignment (ping goes to Unassigned inbox)
47. Reassignment dropdown shows all teams (for managers/owners) or only teams the agent belongs to (for agents)
48. If agent is not member of any team, reassignment dropdown is hidden for agents (managers can still see it)

### Task 1: Database Schema - Teams and Routing

**PURPOSE:** Define database structure for teams, team membership, and routing rules.

- [ ] Create migration: `supabase migration new add_teams_and_routing`
- [ ] Create `agent_teams` table:
  ```sql
  CREATE TABLE IF NOT EXISTS agent_teams (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    tenant_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
    name TEXT NOT NULL,
    description TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    CONSTRAINT agent_teams_unique_name UNIQUE (tenant_id, name)
  );
  ```
- [ ] Create `agent_team_members` table:
  ```sql
  CREATE TABLE IF NOT EXISTS agent_team_members (
    team_id UUID NOT NULL REFERENCES agent_teams(id) ON DELETE CASCADE,
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    added_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (team_id, user_id)
  );
  ```
- [ ] Create `routing_rules` table:
  ```sql
  CREATE TABLE IF NOT EXISTS routing_rules (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    tenant_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
    category_id UUID NOT NULL REFERENCES categories(id) ON DELETE CASCADE,
    rule_type TEXT NOT NULL DEFAULT 'team',
    destination_agent_id UUID REFERENCES users(id) ON DELETE CASCADE,
    destination_team_id UUID REFERENCES agent_teams(id) ON DELETE CASCADE,
    priority INTEGER NOT NULL DEFAULT 0,
    is_active BOOLEAN NOT NULL DEFAULT true,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    CONSTRAINT routing_rules_type_valid CHECK (rule_type IN ('agent', 'team')),
    CONSTRAINT routing_rules_destination_check CHECK (
      (rule_type = 'agent' AND destination_agent_id IS NOT NULL AND destination_team_id IS NULL) OR
      (rule_type = 'team' AND destination_team_id IS NOT NULL AND destination_agent_id IS NULL)
    ),
    CONSTRAINT routing_rules_unique_category UNIQUE (tenant_id, category_id)
  );
  ```
- [ ] Add `team_id` column to pings table:
  ```sql
  ALTER TABLE pings ADD COLUMN team_id UUID REFERENCES agent_teams(id) ON DELETE SET NULL;
  CREATE INDEX idx_pings_team_id ON pings(team_id);
  ```
- [ ] Add indexes:
  - `idx_agent_teams_tenant` on agent_teams(tenant_id)
  - `idx_agent_team_members_user` on agent_team_members(user_id)
  - `idx_routing_rules_tenant_category` on routing_rules(tenant_id, category_id)
- [ ] **Do NOT enable RLS** - tenant isolation at application layer (consistent with other tables)
- [ ] Apply migration: `npx supabase db push --local`

- Routing rules map categories to teams or specific agents
- Agents claim pings from team inbox (sets `assigned_to` while keeping `team_id`)

### Architecture Context

**RLS Note (CRITICAL):**
- RLS is INTENTIONALLY DISABLED on pings, ping_messages, categories, and now teams/routing tables
- Do NOT re-enable RLS on new tables
- Tenant isolation enforced at application layer via auth checks

**Team vs Agent Assignment Model:**
- `pings.team_id` - Which team this ping is routed to (can be null)
- `pings.assigned_to` - Which agent is working on this ping (can be null)
- A ping can have:
  - Neither (unassigned, not routed)
  - Only team_id (routed to team, not yet claimed)
  - Only assigned_to (assigned directly to agent)
  - Both team_id AND assigned_to (agent claimed from team inbox)

**Routing Execution Point:**
The routing happens in `/api/pings/[pingNumber]/echo/continue` endpoint:
1. User completes Echo conversation
2. Echo categorizes the ping (`analysis.readyToSubmit = true`)
3. **NEW:** Check routing rules for the assigned category
4. **NEW:** If rule found, apply routing (set team_id or assigned_to)
5. **NEW:** Create system message about routing
6. Change ping status from 'draft' to 'new'

**Claiming Behavior:**
- Agent clicks "Claim" on unassigned ping in team inbox
- Sets `assigned_to` to agent's ID
- Does NOT change `team_id` (ping stays in team inbox)
- Creates system message "Claimed by {Agent Name}"
- Ping now appears in both team inbox AND "Assigned to Me"

**Reassignment Behavior (Escalation):**
- Agent (ping owner) or manager/owner selects new team from dropdown in ping detail
- `team_id` updated to new team
- `assigned_to` CLEARED (ping becomes unassigned in new team inbox)
- Creates system message "Reassigned from {Old Team} to {New Team}"
- Ping appears in new team inbox, removed from old team inbox
- Ping removed from "Assigned to Me" (agent who reassigned no longer owns it)
- New team member can claim the ping
- **Why clear assigned_to?** The new team may have different expertise; forcing the original agent to stay assigned would defeat the purpose of escalation

- `pings.team_id` - **NEW** UUID reference to agent_teams table
