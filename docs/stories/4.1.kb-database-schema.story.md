**What We're Building:**
This story establishes the database foundation for the Knowledge Base system. The KB will:
1. Auto-generate article drafts from resolved pings (Story 4.2)
2. Allow agents to edit and publish articles (Story 4.3)
3. Enable semantic search via pgvector embeddings (Story 4.4)
4. Provide self-service for end users (Stories 4.5-4.6)

This schema story creates the tables, indexes, and seed data required for all subsequent KB stories.

### RLS Policy - CRITICAL

**DO NOT ENABLE RLS on KB tables.**

Per project convention documented in `CLAUDE.md`, RLS is intentionally disabled on tables requiring Supabase Realtime. Tenant isolation is enforced at the API layer:
1. Authentication verified via Supabase Auth
2. `tenant_id` validated in API routes before queries
3. Admin client (service role) used AFTER validation

This matches the pattern used for `ping_messages`, `pings`, `agent_teams`, and `agent_team_members`.

### Full-Text Search Pattern

Use PostgreSQL tsvector with GIN index for traditional full-text search:

```sql
-- Column
search_vector TSVECTOR GENERATED ALWAYS AS (
  setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
  setweight(to_tsvector('english', coalesce(content, '')), 'B')
) STORED;

-- Index
CREATE INDEX idx_kb_articles_search ON kb_articles USING GIN(search_vector);

-- Query pattern
SELECT * FROM kb_articles
WHERE search_vector @@ plainto_tsquery('english', 'search terms')
ORDER BY ts_rank(search_vector, plainto_tsquery('english', 'search terms')) DESC;
```

### Categories - Reuse Existing Table

**DO NOT create a separate `kb_categories` table.**

KB articles use the existing `categories` table (created in setup wizard) via `category_id` foreign key. This ensures:

- Single source of truth for categories across pings and KB articles
- Categories configured during onboarding automatically available for KB
- Consistent category colors/icons in UI

### Glossary Terms Feature

The `kb_glossary_terms` table enables automatic term linking in KB articles:

**How it works (implemented in future stories):**

1. Agents/managers define terms (acronyms, jargon, company-specific words)
2. When rendering KB article content, match terms against glossary
3. Matched terms become hoverable with tooltip showing definition
4. Example: "Check your SLA compliance" ŌåÆ "SLA" gets tooltip "Service Level Agreement - contractual response/resolution time targets"

**Term categories:**

| Category | Examples |
|----------|----------|
| `acronym` | API, SLA, VPN, DNS, SSO |
| `industry` | Escalation, Triage, Runbook |
| `company` | "Project Phoenix", internal tool names |
| `general` | Technical terms needing explanation |
