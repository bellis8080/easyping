**Why This Story First:**
Many times resolution steps are not documented in the conversation with end users. Agents need a way to document their troubleshooting steps and resolution notes privately. This feature is a prerequisite for Story 4.2.2, which uses these notes to generate high-quality KB articles.

## Story

**As an** agent,
**I want** to add private notes to pings that only other agents can see,
**so that** I can document resolution steps and share context with teammates without cluttering the user conversation.

## Acceptance Criteria

1. Agent message input shows two send options: "Send" (public message to user) and "Private" (internal note)
2. Private notes are stored in `ping_messages` with a visibility flag
3. Private notes are visible to agents, managers, and owners only
4. End users cannot see private notes exist (no indicators, no count, completely hidden)
5. Private notes display with distinct visual styling (e.g., different background color, "Private" badge)
6. When agent resolves a ping without any private notes, Echo prompts: "Add resolution notes for the knowledge base?"
7. Echo's resolution prompt includes a quick-add note input
8. Private notes are included in ping exports and internal reporting

## Dev Notes

### Database Schema Change

```sql
-- Add to ping_messages table
ALTER TABLE ping_messages
ADD COLUMN visibility TEXT NOT NULL DEFAULT 'public';

ALTER TABLE ping_messages
ADD CONSTRAINT ping_messages_visibility_check
CHECK (visibility IN ('public', 'private'));

CREATE INDEX idx_ping_messages_visibility
ON ping_messages(ping_id, visibility);
```

### Message Filtering Pattern

```typescript
// Utility for filtering messages based on user role
export function filterMessagesForUser(
  messages: PingMessage[],
  userRole: UserRole
): PingMessage[] {
  // End users only see public messages
  if (userRole === 'end_user') {
    return messages.filter(msg => msg.visibility === 'public');
  }
  // Agents, managers, owners see all messages
  return messages;
}
```

### Realtime Considerations
[Source: CLAUDE.md - Realtime and RLS section]

RLS is disabled on `ping_messages` for Realtime compatibility. Filtering must happen at:
1. API layer when fetching messages
2. Client-side when receiving realtime updates

For realtime subscriptions, the client should filter incoming messages based on the current user's role before displaying.

### Echo Prompt Logic

The Echo resolution prompt should:
1. Check for existing private notes on the ping
2. If none exist, show the prompt
3. Allow quick note entry without leaving the resolve flow
4. This note provides context for KB article generation (Story 4.2.2)
