**Why This Story:**
Story 4.3.0 allows agents to create and publish KB articles, but users cannot yet browse or discover them. The KB browse page at `/kb` currently uses **mock data**. This story:
- Connects the browse page to real published articles from the database
- Enables category filtering and search
- Prepares the foundation for Story 4.4 (semantic search) and Story 4.5 (article detail page)

**What Already Exists:**
- `/kb` page with full UI layout (header, search bar, category sidebar, article cards, popular articles widget)
- "Manage" button for agents linking to `/kb/manage`
- Mobile-responsive design with filter toggle
- Slate/orange theme styling

**What This Story Adds:**
- Replace mock data with real API calls
- Create public APIs (separate from management APIs)
- Wire up search and category filtering to real data
- Add loading skeletons and proper empty states

## Story

**As a** user or agent,
**I want** to browse the knowledge base with category filtering and see popular articles,
**so that** I can discover relevant support articles and self-serve before creating a ping.

## Acceptance Criteria

1. `/kb` route displays the Knowledge Base browse page accessible from sidebar navigation
2. Published articles displayed as cards showing: title, snippet (first 150 chars of content), category badge, view count, helpfulness percentage, last updated date
3. Categories sidebar (right side) shows all categories with article counts, clicking filters the list
4. "All" category option shows all published articles (default view)
5. Popular Articles widget shows top 5 articles by view count
6. Search bar at top with placeholder "Search articles, guides, and FAQs..."
7. Search uses full-text search with debounced input (300ms)
8. Results show count: "X articles found"
9. Empty state when no articles: "No articles published yet. Check back soon!"
10. Empty search results: "No articles found. Try different keywords or create a ping."
11. Articles sorted by most recently updated by default
12. Clicking an article card navigates to `/kb/[slug]` (detail page, Story 4.5)
13. KB page accessible to all authenticated users (end users, agents, managers, owners)
14. End users see only public content; internal/agent-only articles hidden

### IMPORTANT: RLS is Disabled on Purpose

**DO NOT re-enable Row Level Security (RLS) on `kb_articles` or related tables.**

RLS is intentionally disabled because Supabase Realtime doesn't work properly with RLS enabled. Security is enforced at the API layer:

1. **Authentication:** Verify user via `supabase.auth.getUser()`
2. **Tenant Isolation:** Filter by `tenant_id` from user's profile in ALL queries
3. **Admin Client:** Use `createAdminClient()` only AFTER validation passes

This is documented in CLAUDE.md under "Common Gotchas".

### Search Implementation

For MVP, use `ilike` pattern matching. Story 4.4 adds semantic search with pgvector.

```typescript
// In public articles API
let query = supabaseAdmin
  .from('kb_articles')
  .select(`
    id, title, slug, content, category_id,
    view_count, helpful_count, not_helpful_count,
    updated_at,
    categories(name)
  `, { count: 'exact' })
  .eq('tenant_id', userProfile.tenant_id)
  .eq('status', 'published')
  .is('deleted_at', null);

// Search filter
if (search) {
  query = query.or(`title.ilike.%${search}%,content.ilike.%${search}%`);
}

// Category filter
if (category) {
  // First get category ID by slug
  const { data: cat } = await supabaseAdmin
    .from('kb_categories')
    .select('id')
    .eq('tenant_id', userProfile.tenant_id)
    .eq('slug', category)
    .single();

  if (cat) {
    query = query.eq('category_id', cat.id);
  }
}

query = query.order('updated_at', { ascending: false });
```

### useDebounce Hook

```typescript
// apps/web/src/hooks/use-debounce.ts
import { useState, useEffect } from 'react';

export function useDebounce<T>(value: T, delay: number): T {
  const [debouncedValue, setDebouncedValue] = useState<T>(value);

  useEffect(() => {
    const handler = setTimeout(() => {
      setDebouncedValue(value);
    }, delay);

    return () => clearTimeout(handler);
  }, [value, delay]);

  return debouncedValue;
}
```

### Authorization

Unlike management APIs (which require agent/manager/owner role), public KB APIs are accessible to ALL authenticated users:

```typescript
// No role check - any authenticated user can browse KB
const { data: { user }, error: authError } = await supabase.auth.getUser();

if (authError || !user) {
  return NextResponse.json({ success: false, error: 'Unauthorized' }, { status: 401 });
}

// Get tenant_id for isolation (required)
const { data: userProfile } = await supabase
  .from('users')
  .select('tenant_id')
  .eq('id', user.id)
  .single();
```
