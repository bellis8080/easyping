**What We're Building:**
This story creates the article detail page where users view full KB article content. It builds on:

- The KB browse page that links to `/kb/article/[slug]` (from Story 4.3.5)
- The `kb_articles` table with embeddings (from Stories 4.1 and 4.4)
- The semantic search for finding related articles (from Story 4.4)

**Key Integration Points:**

1. Article detail page at `/kb/article/[slug]` with full markdown rendering
2. "Was this helpful?" feedback widget updates `helpful_count` / `not_helpful_count`
3. Article view tracking in `kb_article_views` table
4. Related articles using semantic similarity from existing hybrid search

## Story

**As a** user,
**I want** to view full KB article content,
**so that** I can self-serve and resolve issues without creating tickets.

## Acceptance Criteria

1. Clicking article from search results or popular articles navigates to article detail page
2. Detail page shows: title, category badge, publish date, author, full content (rendered markdown)
3. "Was this helpful?" feedback widget (Yes/No thumbs up/down)
4. Related articles section at bottom (based on category or semantic similarity)
5. "Still need help? Create a ping" button if article doesn't solve issue
6. Article view tracked in `kb_article_views` table
7. View count displayed on article (e.g., "Viewed 127 times")
8. Breadcrumb navigation: Knowledge Base > Category > Article Title
9. **Agent-only content:** If article has `agent_content`, display it in a separate orange-highlighted section with Lock icon, visible ONLY to agents/managers/owners (hidden from end users)

### Database Schema - KB Articles

The `kb_articles` table structure (actual table name from migrations):

```sql
-- Key columns for this story:
id UUID PRIMARY KEY
tenant_id UUID NOT NULL
title TEXT NOT NULL
slug TEXT NOT NULL
content TEXT NOT NULL  -- Markdown content
category_id UUID REFERENCES categories(id)
author_id UUID REFERENCES users(id)  -- Renamed from created_by
status TEXT  -- 'draft', 'published', 'archived'
view_count INTEGER DEFAULT 0
helpful_count INTEGER DEFAULT 0
not_helpful_count INTEGER DEFAULT 0
published_at TIMESTAMPTZ
updated_at TIMESTAMPTZ
deleted_at TIMESTAMPTZ  -- Soft delete
embedding VECTOR(1536)  -- For semantic similarity
```

**View tracking table** (`kb_article_views`):

```sql
id UUID PRIMARY KEY
article_id UUID REFERENCES kb_articles(id)
user_id UUID REFERENCES users(id)
viewed_at TIMESTAMPTZ DEFAULT NOW()
```

```typescript
// Authentication pattern
const supabase = await createClient();
const { data: { user }, error: authError } = await supabase.auth.getUser();

  return NextResponse.json({ success: false, error: 'Unauthorized' }, { status: 401 });

// Get user's tenant_id
const { data: userProfile } = await supabase
  .from('users')
  .select('tenant_id')
  .eq('id', user.id)
  .single();

const supabaseAdmin = createAdminClient();
// Query with explicit tenant_id filter
.eq('tenant_id', userProfile.tenant_id)
```

### Related Articles Strategy

For finding related articles:

1. **Same category fallback:** Simple query filtering by `category_id`
2. **Semantic similarity (preferred):** Use the article's embedding to find similar articles

```typescript
// If article has embedding, use semantic search
if (article.embedding) {
  const { data: related } = await adminClient.rpc('search_kb_semantic', {
    query_embedding: article.embedding,
    p_tenant_id: tenantId,
    p_limit: 4,  // Get 4 to exclude current article
    p_similarity_threshold: 0.3
  });
  // Filter out current article
  return related.filter(r => r.id !== article.id).slice(0, 3);
}
```
