**What We're Building:**
This story creates the foundation for SLA tracking by implementing the `sla_policies` database table and a management UI for managers/owners to define SLA policies. Each policy maps to a ping priority level (low, normal, high, urgent) with first response and resolution time targets. Default policies are created during first-run setup to ensure every new tenant starts with sensible SLA targets.

**Key Integration Points:**

1. New `sla_policies` table with tenant isolation
2. Replace the placeholder SLA settings page with functional UI
3. API endpoints for CRUD operations on SLA policies
4. Seed default policies during first-run setup
5. Foundation for Story 5.2 (SLA time tracking on pings)

## Story

**As a** manager,
**I want** to define SLA policies with response and resolution time targets,
**so that** I can set service level commitments for my support team.

## Acceptance Criteria

1. `sla_policies` table created with: id, tenant_id, name, priority (matches ping priority), first_response_minutes, resolution_minutes
2. Settings page includes "SLA Policies" section (manager/owner only)
3. Default SLA policies created on setup: Urgent (30min/4hr), High (2hr/24hr), Normal (8hr/3days), Low (24hr/7days)
4. Manager can add/edit/delete SLA policies
5. SLA policy form includes: Name, Priority, First Response Time (minutes), Resolution Time (minutes)
6. SLA policies applied automatically based on ping priority
7. Changes to SLA policies apply to new pings only (existing pings keep original SLA)

## Dev Notes

### Database Schema Reference
[Source: architecture/database-schema.md#sla-policies-table]

The architecture defines the SLA policies table schema:
```sql
CREATE TABLE sla_policies (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  tenant_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  priority TEXT NOT NULL,
  response_time_hours INTEGER NOT NULL,  -- Note: Story uses minutes for finer control
  resolution_time_hours INTEGER NOT NULL,
  is_active BOOLEAN NOT NULL DEFAULT true,
  CONSTRAINT sla_policies_priority_valid CHECK (priority IN ('low', 'normal', 'high', 'urgent')),
  UNIQUE (tenant_id, priority, is_active)
);
```

**Deviation:** The story uses `first_response_minutes` and `resolution_minutes` instead of hours for finer-grained control over SLA targets.

### UI Reference (Wireframes)
[Source: docs/ux/wireframes-mockups.md#screen-3]

The wireframes show SLA timers in the agent inbox:
- Ticket list shows most urgent SLA timer with color badge
- Color coding: Green (>50% time remaining), Yellow (20-50%), Red (<20% or breached)
- Tooltip shows both timers: "✅ First response: 23m (within 2h SLA) | 🟡 Resolution: 1d 14h remaining"

This story creates the policy configuration; Story 5.2 will implement the timer display.

### Existing SLA Page
The placeholder SLA settings page exists at `apps/web/src/app/dashboard/settings/sla/page.tsx`. It currently shows "SLA Management Coming Soon" - this story replaces it with functional content.

### RLS Convention
**IMPORTANT:** Per project convention, RLS is disabled on tables that need realtime updates. The `sla_policies` table should NOT have RLS enabled. Auth is enforced at the API layer.

### Priority Values
Ping priorities are defined in `@easyping/types`:
- `low` | `normal` | `high` | `urgent`

SLA policies map 1:1 with these priorities.

### Time Calculation Example
When a ping is created with priority "high":
1. Look up active SLA policy where priority = 'high'
2. If policy has first_response_minutes = 120, resolution_minutes = 1440
3. Calculate: sla_first_response_due = created_at + 120 minutes
4. Calculate: sla_resolution_due = created_at + 1440 minutes
